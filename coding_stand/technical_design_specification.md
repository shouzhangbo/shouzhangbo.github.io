# 面向 AI Agent 的技术设计文档规范

## 一、概述

本规范旨在为 AI Agent（编程助手）定义一套结构化、可执行的技术设计文档编写标准。规范的核心理念是“规范驱动开发”——让规范成为 AI Agent 与开发者共享的事实来源，确保生成的代码与设计方案保持一致。

本规范面向 AI Agent 执行文档生成任务时使用，要求 Agent 严格按照以下章节结构和规则生成技术设计文档，确保产出的文档结构严谨、逻辑一致、可追溯且便于维护。


## 二、核心原则

### 2.1 规范优先，而非代码优先

在编写任何代码之前，必须先完成设计文档的编写和评审。AI Agent 应优先进入“规划模式”（Plan Mode），先探索现有代码库、起草设计文档，待文档经人工审核确认后再进入编码阶段。

### 2.2 单一事实来源（SSoT）

每一份技术设计文档应成为相关功能模块的权威参考。文档中的每个规则、流程、接口定义都应具有唯一标识，并在全文范围内保持一致的命名和引用方式。

### 2.3 层次化设计

技术设计文档采用分层结构，根据需求复杂度决定产出层级：小需求可合并概要设计与详细设计为单一文档，中型及以上需求则分层产出。

### 2.4 可追溯性

所有技术设计内容必须能追溯到 PRD 中的需求条目。文档版本与 PRD 版本联动，每次需求变更必须同步更新 PRD 和技术设计文档的版本与变更记录。

### 2.5 上下文友好

文档内容应在充分表达设计意图的同时保持精炼，避免信息过载导致 AI Agent 注意力分散或超出上下文窗口限制。


## 三、文档生成的前置条件与决策规则

### 3.1 PRD 文档前置原则

**所有技术设计文档必须以 PRD 文档作为首要输入。** 严禁在缺少 PRD 的情况下生成技术设计文档。Agent 在生成文档前必须执行以下检查：

1. **验证 PRD 存在性**：确认 PRD 文档已就绪，且版本信息完整。
2. **解析 PRD 需求条目**：逐条提取 PRD 中的功能需求（FR）与非功能需求（NFR），并建立需求映射表。
3. **识别需求依赖关系**：分析需求之间的前置依赖与关联关系。

### 3.2 需求规模评估与文档层级决策

Agent 需根据 PRD 的规模和复杂度，自动决定产出的文档层级：

| 需求规模 | 判断标准 | 产出文档 |
|---------|---------|---------|
| **小型需求** | 功能点 ≤ 3 个，影响模块 ≤ 2 个，预计工作量 ≤ 3 人天 | 合并版设计文档（概要与详细合一） |
| **中型需求** | 功能点 3~10 个，影响模块 2~5 个，预计工作量 ≤ 10 人天 | 概要设计 + 详细设计 |
| **大型需求** | 功能点 > 10 个，影响模块 > 5 个，预计工作量 > 10 人天 | 总体设计 + 概要设计 + 详细设计 |

Agent 应在文档开头的“需求规模评估”章节中明确说明规模判断依据及文档层级决策。

### 3.3 UI 文档输入规则

如 PRD 中存在前端需求，Agent 必须检查是否有 UI 设计文档输入：

1. **UI 文档识别**：确认 UI 设计稿（如 Figma 文件）、交互说明文档、组件规范是否存在。
2. **UI 信息提取**：从 UI 文档中提取颜色体系、字体层级、间距规范、组件状态（默认/hover/点击/禁用）等关键信息。
3. **前后端边界划分**：基于 UI 文档明确前端组件与后端接口的职责边界。


## 四、版本管理与变更追踪规范

### 4.1 文档版本命名规范

采用**语义化版本号（SemVer）** 格式：`主版本号.次版本号.修订号`

- **主版本号（Major）** ：重大架构变更、不兼容的接口改动。例如：整体技术栈替换、数据库重构。
- **次版本号（Minor）** ：新增功能模块、新增接口、向前兼容的功能扩展。例如：新增一个业务模块。
- **修订号（Patch）** ：Bug 修复、文档错误更正、小幅优化。例如：修正接口描述、补充边界条件说明。

### 4.2 版本联动机制

**技术设计文档的版本必须与 PRD 文档的版本联动。** 具体规则如下：

| PRD 变更类型 | 技术设计文档操作 | 版本变更 |
|------------|----------------|---------|
| 新增功能需求 | 新增对应的设计章节 | 次版本号 +1 |
| 修改/删除现有需求 | 修改或删除对应设计内容，记录变更原因 | 次版本号 +1 |
| 仅修正 PRD 文字/格式 | 无需变更设计文档 | 不变更 |
| 架构级重大调整 | 重新设计并标注废弃内容 | 主版本号 +1 |

### 4.3 版本修订记录表（必填）

每份文档必须包含以下格式的版本修订记录表：

| 版本号 | 修订日期 | 修订人 | 关联 PRD 版本 | 变更类型 | 变更描述 | 影响范围 |
|-------|---------|-------|-------------|---------|---------|---------|
| V1.0.0 | YYYY-MM-DD | XXX | PRD V1.2.0 | 新建 | 初始版本 | — |
| V1.1.0 | YYYY-MM-DD | XXX | PRD V1.3.0 | 新增功能 | 新增用户权限模块 | 第 5 章、第 7 章 |
| V1.1.1 | YYYY-MM-DD | XXX | PRD V1.3.0 | 修正 | 修正接口参数类型描述 | 第 8.2 节 |

**变更类型**取值为：新建、新增功能、功能修改、功能删除、Bug 修复、文档优化、架构调整。

### 4.4 需求条目级版本追踪

对于大型项目，建议对每个需求条目单独记录版本变更。技术设计文档中引用的每个 PRD 需求条目（以 FR-XXX 标识）都应在文档中标注其来源版本。

**示例**：
> 本模块实现的功能对应 **PRD FR-012（来源：PRD V1.3.0）** 和 **PRD FR-015（来源：PRD V1.3.2）**。

### 4.5 变更影响范围标识

每次文档修订必须在修订记录中明确标注**变更影响范围**，便于开发、测试团队快速定位受影响的章节和模块。影响范围建议以“章节号”或“模块名”的形式列出。


## 五、UI 文档与前端技术设计的关联规则

### 5.1 UI 设计文档的必提取信息

当存在 UI 设计文档时，Agent 必须从前端技术设计章节中体现以下内容：

#### 5.1.1 视觉规范映射

| UI 规范项 | 前端设计对应项 | 示例 |
|----------|--------------|------|
| 主色/辅助色/中性色 | CSS 变量定义 | `--color-primary: #165DFF;` |
| 字体层级（标题/正文/注释） | 全局样式定义 | `.text-title-lg { font-size: 24px; }` |
| 按钮控件规范 | 组件 Props 定义 | `<Button variant="primary" />` |
| 间距规范 | 设计 Token 映射 | `--spacing-md: 16px;` |
| 圆角/阴影 | 全局样式变量 | `--radius-card: 8px;` |

#### 5.1.2 组件状态设计

从 UI 文档中提取每个组件的状态定义，并在详细设计中明确实现方式：

- 默认状态（default）
- 悬停状态（hover）
- 激活/点击状态（active）
- 禁用状态（disabled）
- 加载状态（loading）
- 错误状态（error）

#### 5.1.3 交互逻辑说明

从交互说明文档中提取以下信息并转化为前端实现方案：

- 页面跳转流程与路由定义
- 组件间通信方式（事件总线/Props 传递/状态管理）
- 数据加载与刷新机制
- 异常状态处理（无数据、加载失败、网络异常等）

### 5.2 字段极值处理规则

前端设计必须处理 UI 文档中未明确的边界情况，Agent 应主动补充：

- 文本超长处理方案（截断 + Tooltip / 换行 / 滚动）
- 数据为空时的占位展示
- 极端数据量下的性能考虑（如列表虚拟滚动）

### 5.3 响应式与适配规则

如 UI 文档提供了多端设计，Agent 需明确：

- 断点定义与布局切换策略
- 移动端/桌面端差异化交互
- 单位转换规则（px → rem/vw/vh）


## 六、技术设计文档层级与模板

### 6.1 文档层级定义

| 层级 | 名称 | 核心问题 | 适用场景 |
|-----|------|---------|---------|
| L3 | 总体设计 | 系统整体架构怎么搭？技术怎么选？ | 大型需求/新项目启动 |
| L4 | 概要设计 | 系统由哪些模块组成？模块之间如何关联？ | 中型及以上需求 |
| L5 | 详细设计 | 每个模块具体如何实现？接口如何定义？ | 所有需求 |

概要设计关注系统的宏观结构——解决“做什么”和“如何连接”的问题；详细设计深入模块内部——解决“具体怎么做”的问题。

## 七、AI Agent 生成文档时的行为规范

### 7.1 文档生成流程

Agent 生成技术设计文档必须严格遵循以下流程：

1. **输入验证**：检查 PRD 是否存在且版本有效；检查 UI 文档是否存在（如有前端需求）。
2. **需求解析**：逐条提取 PRD 功能需求，生成需求映射表。
3. **规模评估**：根据需求规模决定文档层级（合并版/概要+详细/总体+概要+详细）。
4. **文档生成**：按对应模板生成文档内容，确保每个章节有实质内容。
5. **版本初始化**：设置初始版本号 V1.0.0，填写版本修订记录。
6. **交叉引用检查**：确保文档中所有对 PRD 和 UI 文档的引用准确。
7. **自检**：在交付前完成自检清单中的所有检查项。

### 7.2 自检清单

Agent 在生成文档后、交付前必须执行以下自检：

| # | 检查项 | 通过标准 |
|---|-------|---------|
| 1 | PRD 依赖检查 | 文档中明确标注了关联 PRD 的版本号 |
| 2 | 需求映射检查 | 每个设计功能点都能追溯到 PRD 需求条目 |
| 3 | UI 关联检查 | 前端设计章节引用了 UI 文档中的组件和规范 |
| 4 | 版本记录检查 | 版本修订记录表已填写，格式正确 |
| 5 | 规模决策检查 | 已说明需求规模评估结果及文档层级决策理由 |
| 6 | 接口完整性检查 | 每个接口定义了完整的请求/响应参数和错误码 |
| 7 | 异常处理检查 | 覆盖了主要异常场景及处理方式 |
| 8 | 影响范围检查 | 标注了对现有模块的影响范围 |
| 9 | 引用完整性检查 | 所有文档内引用和外部引用均可追溯 |
| 10 | 模板完整性检查 | 各章节无缺失，无空占位符 |

### 7.3 禁止行为

Agent 在生成文档时严禁以下行为：

- ❌ 在 PRD 缺失或 PRD 版本无效的情况下生成技术设计文档。
- ❌ 在有 UI 文档的情况下忽略 UI 信息，生成与设计稿不一致的前端设计。
- ❌ 跳过规模评估，盲目套用模板。
- ❌ 生成空洞的章节内容（如“待补充”、“略”等占位语）。
- ❌ 忽略 PRD 更新，仍基于旧版本 PRD 生成或更新文档。
- ❌ 在更新文档时遗漏修改版本修订记录。
- ❌ 生成的技术方案与 PRD 需求无追溯关系。


## 八、文档评审与交付标准

### 8.1 交付物清单

| 文档类型 | 适用条件 | 交付物 |
|---------|---------|-------|
| 小型需求 | 功能点 ≤ 3 个 | 1 份合并版设计文档 |
| 中型需求 | 功能点 3~10 个 | 1 份概要设计 + N 份详细设计 |
| 大型需求 | 功能点 > 10 个 | 1 份总体设计 + 1 份概要设计 + N 份详细设计 |

### 8.2 文档质量标准

- **完整性**：所有必填章节均已填写，内容充实，无占位符。
- **一致性**：文档内部无矛盾，与 PRD 和 UI 文档保持一致。
- **可追溯性**：设计内容可追溯到 PRD 需求条目。
- **可执行性**：详细设计的内容可直接指导编码，不依赖额外的口头说明。
- **版本可追溯**：版本修订记录完整，变更影响范围清晰。
