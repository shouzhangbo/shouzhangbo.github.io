# MySQL 字段类型、存储引擎、索引、锁与分布式架构面试资料

## 1. MySQL 字段类型详解

MySQL 字段类型可以简单分为三大类：**数值类型**、**字符串类型**和**日期时间类型**。

### 1.1 数值类型

| 类型 | 大小（字节） | 范围（有符号） | 范围（无符号） | 用途 |
|------|-------------|---------------|---------------|------|
| TINYINT | 1 | (-128, 127) | (0, 255) | 小整数值 |
| SMALLINT | 2 | (-32768, 32767) | (0, 65535) | 中等整数值 |
| MEDIUMINT | 3 | (-8388608, 8388607) | (0, 16777215) | 较大整数值 |
| INT | 4 | (-2147483648, 2147483647) | (0, 4294967295) | 标准整数值 |
| BIGINT | 8 | (-2^63, 2^63-1) | (0, 2^64-1) | 极大整数值 |
| FLOAT | 4 | 单精度浮点数 | - | 近似小数 |
| DOUBLE | 8 | 双精度浮点数 | - | 近似小数 |
| DECIMAL(M,D) | 变长 | 依赖M和D | 依赖M和D | 精确小数 |

**关键知识点**：
- **UNSIGNED 修饰符**：规定字段只保存正值，MySQL字段默认是有符号的。
- **ZEROFILL**：指定后MySQL会自动为列加上UNSIGNED属性，并在显示时用零填充。
- **DECIMAL 与 FLOAT/DOUBLE 的区别**：DECIMAL是**定点数**，可以存储精确的小数值；FLOAT/DOUBLE是**浮点数**，只能存储近似值。DECIMAL用于存储有精度要求的小数（如金钱相关数据）。

### 1.2 字符串类型

| 类型 | 说明 | 最大长度 | 特点 |
|------|------|---------|------|
| CHAR(M) | 定长字符串 | M字符 | 固定长度，检索快，可能浪费空间 |
| VARCHAR(M) | 变长字符串 | M字符 | 节省空间，需额外1-2字节记录长度 |
| TINYTEXT | 短文本 | 255字节 | 小型文本 |
| TEXT | 文本 | 64KB | 常规文本 |
| MEDIUMTEXT | 中等文本 | 16MB | 较大文本数据 |
| LONGTEXT | 长文本 | 4GB | 极大文本数据 |
| TINYBLOB | 二进制短数据 | 255字节 | 小型二进制 |
| BLOB | 二进制数据 | 64KB | 常规二进制 |
| MEDIUMBLOB | 中等二进制 | 16MB | 较大二进制数据 |
| LONGBLOB | 长二进制 | 4GB | 极大二进制数据 |

#### CHAR 与 VARCHAR 的区别
- **CHAR是定长字符串**：存储时在右边填充空格以达到指定长度，检索时会去掉空格。
- **VARCHAR是变长字符串**：存储时使用1或2个额外字节记录字符串长度，检索时不需要处理。
- **适用场景**：CHAR适合存储长度较短或长度都差不多的字符串（如身份证号码、MD5加密后的密码）；VARCHAR适合存储长度不确定或差异较大的字符串（如用户昵称、文章标题）。

#### varchar(100) 和 varchar(10) 的区别
- 两者都是变长类型，表示能存储最多100个字符和10个字符。
- **存储相同字符串时，占用磁盘空间相同**（这是很多人容易误解的一点）。
- **内存消耗不同**：varchar在内存中操作时，通常会分配固定大小的内存块，即使用字符类型中定义的长度。varchar(100)会消耗更多内存。

#### 为什么不推荐使用 TEXT 和 BLOB
- 不能有默认值。
- 在遇到使用临时表的情况时，无法使用内存临时表，只能在磁盘上创建临时表。
- 检索效率比 CHAR 和 VARCHAR 低。
- 不能直接创建索引，需要指定前缀长度。
- 会消耗大量的网络和IO带宽。
- 可能导致表上的DML操作都变得较慢。

### 1.3 日期时间类型

| 类型 | 大小 | 范围 | 格式 | 用途 |
|------|-----|------|------|------|
| YEAR | 1字节 | 1901-2155 | YYYY | 年份 |
| TIME | 3字节 | -838:59:59 到 838:59:59 | HH:MM:SS | 时间值或持续时间 |
| DATE | 3字节 | 1000-01-01 到 9999-12-31 | YYYY-MM-DD | 日期值 |
| DATETIME | 8字节 | 1000-01-01 00:00:00 到 9999-12-31 23:59:59 | YYYY-MM-DD HH:MM:SS | 混合日期和时间值 |
| TIMESTAMP | 4字节 | 1970-01-01 00:00:01 到 2038-01-19 03:14:07 | YYYY-MM-DD HH:MM:SS | 时间戳，与时区相关 |

**TIMESTAMP特性**：如果对TIMESTAMP字段没有明确赋值，或被赋予null值，MySQL会自动使用系统当前的日期和时间来填充。

## 2. 存储引擎

### 2.1 存储引擎概览

MySQL 5.5及之后版本默认使用 **InnoDB** 作为存储引擎。

### 2.2 InnoDB 引擎

**核心特点**：
- 支持**事务**（ACID），默认隔离级别为 **REPEATABLE-READ** 且能解决幻读问题
- 支持**行级锁**，使用**多版本并发控制（MVCC）**获取高并发性
- 支持**外键**约束
- 使用 **B+树索引**结构
- 实现了SQL标准的4种隔离级别，并使用 **next-key locking** 策略避免幻读现象
- 支持**崩溃恢复**能力

**高并发特性**：
- 插入缓冲（insert buffer）
- 二次写（double write）
- 自适应哈希索引（adaptive hash index）
- 预读（read ahead）

**默认特性**：在MySQL 5.6及以上的版本中，官方不再支持MyISAM引擎和Memory引擎，主要因为InnoDB具有明显的性能优势。

### 2.3 MyISAM 引擎

**核心特点**：
- **不支持事务**
- 采用**表级锁**机制
- 表由 `.MYD`（数据文件）和 `.MYI`（索引文件）组成
- 只缓存索引数据，不缓存数据
- MySQL 5.0以后默认支持256TB的表数据

**适用场景**：适合OLAP应用，读密集型场景。

**缺陷**：对数据完整性的保护存在缺陷，可能导致数据损坏甚至丢失，且这些缺陷很多是设计问题，无法在不破坏兼容性的前提下修复。

### 2.4 其他存储引擎

| 引擎 | 特点 | 适用场景 |
|------|------|---------|
| Memory | 数据放在内存，默认使用hash索引，不支持TEXT/BLOB类型 | 临时表、缓存 |
| NDB | 数据放在内存（MySQL 5.1后可放部分索引到磁盘），支持高可用集群 | 电信级高并发应用 |
| Archive | 压缩能力较强 | 日志归档 |
| RocksDB | LSM Tree结构，存储空间比InnoDB节省一半以上 | 数据量大、写密集型场景 |
| LibraDB | 只读分析引擎，列式存储，支持MPP并行计算 | 复杂分析查询 |

### 2.5 各引擎对比

| 特性 | InnoDB | MyISAM | Memory |
|------|--------|--------|--------|
| 事务支持 | ✔ | ✘ | ✘ |
| 锁粒度 | 行锁 | 表锁 | 表锁 |
| 外键支持 | ✔ | ✘ | ✘ |
| MVCC | ✔ | ✘ | ✘ |
| 崩溃恢复 | ✔ | ✘ | ✘ |
| 索引类型 | B+树 | B+树 | Hash/B+树 |
| 数据持久化 | ✔ | ✔ | ✘（重启丢失） |

## 3. 索引原理与优化

### 3.1 索引数据结构

#### B-Tree 与 B+Tree 的区别

**B-Tree特点**：
- 非叶子节点也保存具体数据
- 查找某个关键字时找到即可返回
- 不利于范围查询

**B+Tree特点（InnoDB使用）**：
- 非叶子节点只存储关键字信息（索引），不存储数据
- 叶子节点存储所有数据，并通过链表相连
- 任何关键字的查找都必须走从根节点到叶子节点的路径，查询效率稳定

**为什么InnoDB选择B+Tree**：

| 对比项 | B-Tree | B+Tree | 优势 |
|--------|--------|--------|------|
| 非叶子节点存储 | 关键字+数据 | 仅关键字 | B+树同一节点能存储更多关键字，降低树高 |
| 叶子节点结构 | 分散 | 链表连接 | B+树支持高效范围查询 |
| 查询效率 | 不稳定（可能在非叶子节点找到） | 稳定（必须到叶子节点） | B+树所有数据查询效率相当 |
| 磁盘IO次数 | 树高更大，IO更多 | 树高更小，IO更少 | B+树磁盘读写代价更低 |

**B+树优势总结**：
1. **磁盘读写代价更低**：内部节点不存储数据，同一盘块可容纳更多关键字，减少IO次数
2. **查询效率更稳定**：任何关键字的查找路径长度相同
3. **更利于范围查询**：叶子节点通过链表连接，找到起始值后可直接遍历

### 3.2 索引分类

#### 聚簇索引
- **定义**：数据行的物理存储顺序与索引顺序相同
- **特点**：
  - 叶子节点存储完整的用户记录
  - 一个表只能有一个聚簇索引
  - InnoDB中，主键列作为聚簇索引

#### 二级索引（非聚簇索引）
- **定义**：叶子节点存储索引列的值和对应的主键值
- **特点**：
  - 一个表可以有多个二级索引
  - 需要通过主键值再去聚簇索引查找完整记录（回表）

#### 覆盖索引
- **定义**：查询的所有列都包含在索引中，可以直接从索引获取数据，避免回表
- **示例**：
  ```sql
  CREATE INDEX idx_name_age ON users(name, age);
  SELECT name, age FROM users WHERE name='张三'; -- 使用覆盖索引
  ```

#### 前缀索引
- **定义**：对字符串列的前N个字符建立索引
- **适用场景**：长字符串列，保证区分度的前提下减少索引大小

### 3.3 索引设计原则

#### 最左前缀原则
- **核心**：联合索引必须从最左列开始使用，如果跳过最左列，索引将完全失效
- **规则**：对于索引`(a,b,c)`：
  - ✅ 支持：`WHERE a=1`、`WHERE a=1 AND b=2`、`WHERE a=1 AND b=2 AND c=3`、`WHERE a=1 AND c=3`（只能用到a）
  - ❌ 不支持：`WHERE b=2`、`WHERE c=3`、`WHERE b=2 AND c=3`

#### 选择性原则
- **定义**：不重复的索引值与记录总数的比值
- **公式**：`COUNT(DISTINCT column_name) / COUNT(*)`
- **建议**：选择选择性大于0.1的列作为索引

#### 最小化原则
- 控制单表索引数量，一般不超过5个
- 组合索引优于单列索引
- 避免重复或冗余索引

#### 频率原则
- 高频查询列优先建立索引
- 频繁更新的列谨慎建立索引

#### 长度原则
- 对字符串列建立索引时，考虑前缀索引
- 在保证区分度的前提下选择更短的索引

### 3.4 索引失效场景

| 场景 | 示例 | 解决方案 |
|------|------|---------|
| 违反最左前缀原则 | `WHERE age > 25`（联合索引`(name,age)`） | 调整查询顺序或修改索引顺序 |
| 索引列参与表达式计算 | `WHERE DATE_FORMAT(create_time,'%Y-%m-%d') = '2023-08-01'` | 改用范围查询或使用生成列 |
| 隐式类型转换 | `WHERE id = 10086`（id为varchar类型） | 保持字段类型与查询值类型一致 |
| OR连接非索引字段 | `WHERE name = '王伟' OR department = '技术部'`（department无索引） | 拆分查询使用UNION或建立全覆盖索引 |
| 低选择性索引被优化器放弃 | `SELECT * FROM members WHERE gender = 'M'`（gender区分度低） | 组合高区分度字段或使用覆盖索引 |
| 使用不等于操作 | `WHERE status != 'active'` | 考虑业务逻辑改写 |
| LIKE以通配符开头 | `WHERE name LIKE '%张三'` | 避免前导通配符 |

### 3.5 回表与索引下推

#### 回表原理
- **定义**：通过二级索引查询时，需要再到聚簇索引中获取完整行记录的过程
- **过程**：
  1. 通过二级索引B+树查找，找到满足条件的主键值
  2. 通过主键值再去聚簇索引查找对应的完整行记录
- **影响**：额外的磁盘IO开销，结果集越大影响越显著

#### 索引下推（ICP）
- **定义**：MySQL 5.6引入的优化技术，在存储引擎层过滤不满足条件的记录
- **原理**：只有满足条件的记录才会被返回给服务器层，减少回表次数和数据传输量
- **适用场景**：二级索引无法完全覆盖查询，且有多个过滤条件可在索引中判断

## 4. 查询优化

### 4.1 EXPLAIN 执行计划

**关键字段含义**：

| 字段 | 含义 | 优化目标 |
|------|------|---------|
| type | 访问类型（const > eq_ref > ref > range > index > ALL） | 尽量达到range或ref |
| key | 实际使用的索引 | 确保索引被使用 |
| key_len | 使用的索引长度 | 越短越好 |
| rows | 预估需要扫描的行数 | 越小越好 |
| Extra | 额外信息（Using index、Using where、Using filesort等） | 避免filesort、temporary |

### 4.2 慢查询优化步骤
1. 开启慢查询日志，定位问题SQL
2. 使用EXPLAIN分析执行计划
3. 检查索引是否合理（是否失效、是否覆盖）
4. 检查SQL写法（避免函数、隐式转换等）
5. 检查表结构（字段类型、是否分表）
6. 分批处理大查询

### 4.3 分页优化

**大偏移量问题**：
```sql
-- 不推荐（偏移量大时性能差）
SELECT * FROM users ORDER BY id LIMIT 100000, 10;

-- 优化方案1：记录上次位置
SELECT * FROM users WHERE id > 100000 ORDER BY id LIMIT 10;

-- 优化方案2：延迟关联
SELECT u.* FROM users u 
INNER JOIN (SELECT id FROM users ORDER BY id LIMIT 100000, 10) tmp 
ON u.id = tmp.id;
```

## 5. 锁机制

### 5.1 锁的分类

MySQL的锁可以从三个维度进行分类：

| 分类维度 | 锁类型 | 说明 |
|---------|-------|------|
| 态度 | 悲观锁 | 认为数据总会冲突，操作前加锁 |
| 态度 | 乐观锁 | 认为数据不会冲突，通过版本号机制实现 |
| 类型 | 读锁（共享锁） | 可同时加多个，互不影响 |
| 类型 | 写锁（排他锁） | 阻塞其他读锁和写锁 |
| 粒度 | 表锁 | 锁整张表，开销小粒度大 |
| 粒度 | 行锁 | 锁单行，开销大粒度小 |
| 粒度 | 页面锁 | 介于表锁和行锁之间 |
| 特殊 | 间隙锁 | 锁间隙，防止幻读 |
| 特殊 | 临键锁 | 行锁+间隙锁 |

### 5.2 悲观锁与乐观锁

#### 悲观锁
- **实现**：依赖数据库提供的锁机制（`SELECT ... FOR UPDATE`）
- **特点**：在整个数据处理过程中将数据锁定

#### 乐观锁
- **实现**：通过数据版本记录机制（MVCC是乐观锁的一种实现方式）
- **常见方式**：增加version版本号字段，更新时对比版本号

### 5.3 表锁
- **表共享锁**：加锁后其他线程可读不可写
- **表独占写锁**：加锁后只有当前线程可读写
- **命令**：
  ```sql
  LOCK TABLE user_table READ;  -- 加读锁
  LOCK TABLE user_table WRITE; -- 加写锁
  UNLOCK TABLES;               -- 释放锁
  ```

### 5.4 行锁
- **特点**：InnoDB行锁针对**索引**加锁，不是针对记录
- **重要**：如果对非索引字段设置条件进行更新，行锁可能会变成表锁
- **命令**：
  ```sql
  SELECT * FROM user WHERE id = 1 LOCK IN SHARE MODE; -- 共享锁
  SELECT * FROM user WHERE id = 1 FOR UPDATE;         -- 排他锁
  ```

### 5.5 间隙锁与临键锁

#### 间隙锁（Gap Lock）
- **定义**：在范围查询时，对条件范围内并不存在的记录间隙加锁
- **作用**：防止其他事务在间隙中插入数据
- **生效条件**：只在**可重复读**隔离级别下生效

#### 临键锁（Next-Key Lock）
- **定义**：行锁和间隙锁的组合，是一种特殊的间隙锁
- **特点**：锁住一段**左开右闭区间**的数据
- **作用**：解决幻读问题
- **注意**：只与非唯一索引列有关，在唯一索引列（包括主键列）上不存在临键锁

**示例**：
假设表中有id=1,2,3,7,10,21的记录。
执行`UPDATE user SET sex=1 WHERE id>8 AND id<18;`
会锁住(7,10]、(10,21]这两个间隙，其他事务无法在这个区间内插入或修改数据。

### 5.6 MVCC 原理
- **全称**：Multi-Version Concurrency Control（多版本并发控制）
- **作用**：在**可重复读**隔离级别下解决幻读问题
- **实现**：通过隐藏字段（DB_TRX_ID、DB_ROLL_PTR）和undo日志实现多版本链

## 6. 事务与隔离级别

### 6.1 ACID 特性

| 特性 | 说明 | 实现机制 |
|------|------|---------|
| 原子性 | 事务要么全成功，要么全失败 | undo log |
| 一致性 | 事务执行前后数据保持一致 | 代码约束+数据库约束 |
| 隔离性 | 并发事务之间互不干扰 | 锁+MVCC |
| 持久性 | 事务提交后永久保存 | redo log |

### 6.2 隔离级别

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 实现方式 |
|---------|------|-----------|------|---------|
| 读未提交 | 可能 | 可能 | 可能 | 不加锁 |
| 读已提交 | 不可能 | 可能 | 可能 | 每次读都重新生成ReadView |
| 可重复读（默认） | 不可能 | 不可能 | 可能（InnoDB可解决） | 事务开始时生成ReadView + next-key锁 |
| 串行化 | 不可能 | 不可能 | 不可能 | 所有读都加锁 |

**InnoDB的可重复读**：通过MVCC和next-key locking策略，可以解决幻读问题。

## 7. 分布式MySQL架构

### 7.1 主从复制

**基本原理**：
1. 主库将数据变更写入二进制日志（binlog）
2. 从库的IO线程读取主库的binlog并写入中继日志（relay log）
3. 从库的SQL线程重放中继日志，实现数据同步

**复制类型**：
- **异步复制**：主库不等待从库确认
- **半同步复制**：至少一个从库确认写入成功后再返回主库
- **同步复制**：所有从库都确认

**常见问题**：
- **主从延迟**：可通过并行复制、分库分表优化
- **数据一致性**：GTID（全局事务标识符）确保复制位置准确

### 7.2 读写分离

**实现方式**：
- **应用层逻辑**：通过中间件（MyCat、ShardingSphere）实现
- **客户端直连**：配置多个数据源，应用层决定读写路由

**注意事项**：
- 主从延迟可能导致读写不一致，可强制读主库或等待延迟消除

### 7.3 分库分表

#### 垂直拆分
- **垂直分库**：按业务模块拆分到不同数据库
- **垂直分表**：将大表的列拆分到多个表（如将text/blob字段拆分）

#### 水平拆分
- **范围分片**：按id范围划分（如1-1000在分片1，1001-2000在分片2）
- **哈希分片**：按hash值取模划分
- **一致性哈希**：减少节点变化时的数据迁移

**分片键选择**：尽量选择查询频率高的列，避免跨分片查询。

### 7.4 分布式ID生成

- **雪花算法**：64位长整型，由时间戳+机器ID+序列号组成
- **数据库号段模式**：批量获取ID号段
- **UUID**：性能差、无序，不推荐

### 7.5 高可用架构

| 方案 | 特点 | 适用场景 |
|------|------|---------|
| MHA | 自动故障切换，30秒内完成切换 | 主从架构 |
| MMM | 双主互备，自动切换 | 双主架构 |
| InnoDB Cluster | MySQL官方高可用方案，基于Group Replication | 强一致性要求 |
| 异地多活 | 多地域部署，负载均衡 | 超大规模、容灾要求高 |

**异地多活核心挑战**：
- 数据同步延迟
- 网络不稳定
- 数据一致性维护
- 系统监控复杂性

**解决方案**：
- 使用半同步复制或PXC实现低延迟同步
- 部署网络监控工具，优化网络架构
- 使用分布式事务管理器或补偿机制（如Saga模式）
- 部署自动化监控工具
