# JVM 内存、GC、调优与排查问题部分面试资料

## 1. JVM 内存区域划分

### 1.1 运行时数据区概览
Java虚拟机在执行Java程序时会将其管理的内存划分为若干个不同的数据区域，这些区域统称为**运行时数据区**。根据是否线程共享，可分为两类：

- **线程私有区域**：随线程创建而创建，随线程销毁而销毁。
  - 程序计数器
  - Java虚拟机栈
  - 本地方法栈
- **线程共享区域**：随虚拟机启动而创建，随虚拟机退出而销毁。
  - Java堆
  - 方法区（元空间）
  - 运行时常量池（方法区的一部分）

### 1.2 程序计数器
- **定义**：一块较小的内存空间，可以看作是当前线程所执行的字节码的行号指示器。
- **作用**：通过改变计数器的值来选取下一条需要执行的字节码指令，分支、循环、跳转、异常处理、线程恢复等基础功能都需要依赖计数器完成。
- **特点**：
  - 线程私有，每个线程都有独立的程序计数器。
  - 如果线程执行的是Java方法，计数器记录的是正在执行的虚拟机字节码指令地址；如果执行的是Native方法，计数器值为空（Undefined）。
  - 此内存区域是唯一一个在Java虚拟机规范中没有规定任何`OutOfMemoryError`情况的区域。

### 1.3 Java虚拟机栈
- **定义**：线程私有，生命周期与线程相同。描述的是Java方法执行的线程内存模型。
- **栈帧**：每个方法执行时都会创建一个**栈帧**，用于存储局部变量表、操作数栈、动态连接、方法出口等信息。一个方法的调用到执行完成，对应一个栈帧在虚拟机栈中入栈到出栈的过程。
  - **局部变量表**：存放方法参数和方法内部定义的局部变量。编译期间完成分配，运行期间大小不变。
  - **操作数栈**：用于存放计算过程中的临时数据，是一个后进先出栈。
  - **动态连接**：将符号引用转换为直接引用。
- **异常**：
  - `StackOverflowError`：线程请求的栈深度大于虚拟机允许的深度（如递归过深）。
  - `OutOfMemoryError`：虚拟机栈支持动态扩展，如果扩展时无法申请到足够内存（如创建过多线程）。

### 1.4 本地方法栈
- **定义**：与虚拟机栈类似，区别在于虚拟机栈为虚拟机执行Java方法服务，而本地方法栈为虚拟机使用到的**Native方法**服务。
- **异常**：同样抛出`StackOverflowError`和`OutOfMemoryError`。

### 1.5 Java堆
- **定义**：虚拟机启动时创建，是所有线程共享的内存区域，几乎所有的对象实例和数组都在这里分配内存。
- **地位**：垃圾收集器管理的主要区域，因此也称为**GC堆**。
- **分代划分**（逻辑划分，G1等收集器已物理上不分代但逻辑上保留）：
  - **新生代**：对象诞生和成长的地方，大部分对象会被回收。进一步细分为Eden区和两个Survivor区（From、To）。
  - **老年代**：存放生命周期长的对象，如从新生代晋升的对象。
- **配置参数**：
  - `-Xms`：堆初始大小。
  - `-Xmx`：堆最大大小。生产环境通常将两者设置为相同值，避免运行时动态扩容带来的性能损耗。
  - `-Xmn`：新生代大小。
  - `-XX:SurvivorRatio`：Eden区与Survivor区的比例，默认8:1:1。
  - `-XX:NewRatio`：新生代与老年代的比例。

### 1.6 方法区（元空间）
- **定义**：线程共享，用于存储已被虚拟机加载的**类型信息、常量、静态变量、即时编译器编译后的代码缓存**等数据。
- **演变**：
  - JDK 1.7及以前：方法区通过**永久代（PermGen）**实现。
  - JDK 1.8及以后：永久代被移除，替换为**元空间（Metaspace）**，存储在本地内存中。
- **元空间优势**：使用本地内存，默认大小仅受本地内存限制，减少了`OutOfMemoryError: PermGen`的常见问题。
- **运行时常量池**：方法区的一部分，用于存放编译期生成的各种字面量和符号引用。
- **配置参数**：
  - `-XX:MetaspaceSize`：元空间初始大小。
  - `-XX:MaxMetaspaceSize`：元空间最大大小。
  - `-XX:+UseCompressedOops`：启用压缩指针，减少内存占用。
  - `-XX:+UseCompressedClassPointers`：使用压缩类指针。

### 1.7 直接内存
- **定义**：不是虚拟机运行时数据区的一部分，也不是Java虚拟机规范中定义的内存区域。但在NIO中频繁使用，通过堆上的`DirectByteBuffer`对象操作直接内存。
- **特点**：避免了在Java堆和Native堆之间来回复制数据，提高性能。
- **异常**：可能抛出`OutOfMemoryError`。

## 2. 对象存活判定与垃圾回收基础

### 2.1 判断对象是否可回收

#### 2.1.1 引用计数法
- **原理**：为每个对象维护一个引用计数器，当有一个地方引用它时计数器+1，引用失效时-1。计数器为0的对象可回收。
- **缺点**：无法解决**循环引用**问题，主流Java虚拟机未采用。

#### 2.1.2 可达性分析算法
- **原理**：通过一系列称为**GC Roots**的对象作为起点，从这些节点开始向下搜索，搜索走过的路径称为**引用链**。当一个对象到GC Roots没有任何引用链相连（即不可达），则证明此对象可回收。
- **可作为GC Roots的对象**：
  1. 虚拟机栈（栈帧中的本地变量表）中引用的对象。
  2. 方法区中类静态属性引用的对象。
  3. 方法区中常量引用的对象。
  4. 本地方法栈中JNI（Native方法）引用的对象。
  5. Java虚拟机内部的引用（如基本数据类型对应的Class对象、常驻异常对象等）。
  6. 所有被同步锁（synchronized）持有的对象。
  7. 反映Java虚拟机内部情况的JMXBean、JVMTI回调等。

#### 2.1.3 对象"死亡"的两次标记过程
- 即使在可达性分析中不可达的对象，也并非"非死不可"。要真正宣告一个对象死亡，至少要经历**两次标记**：
  1. **第一次标记**：可达性分析发现对象不可达，进行第一次标记并筛选，判断是否有必要执行`finalize()`方法。
  2. **第二次标记**：如果对象被判定为有必要执行`finalize()`，会将其放入一个队列，由虚拟机自动创建一个低优先级线程执行。如果在`finalize()`中对象与引用链上的任何对象建立关联，则会被移出"即将回收"集合。否则，真正回收。
- **注意**：`finalize()`在JDK 9已被弃用，不应依赖它来释放资源。

### 2.2 引用类型
JDK 1.2之后，Java将引用分为四种，引用强度依次减弱：

| 引用类型 | 回收时机 | 常见用途 |
|---------|---------|---------|
| **强引用** | 永不回收，即使OOM | `Object obj = new Object()` |
| **软引用** | 内存不足时回收 | 缓存实现（如`SoftReference`） |
| **弱引用** | 下次垃圾回收时回收 | `WeakHashMap`、`ThreadLocal`的key |
| **虚引用** | 任何时候都可能回收，无法通过它获取对象实例 | 对象回收跟踪（如`PhantomReference`配合引用队列） |

### 2.3 方法区的回收
- **废弃常量回收**：与对象回收类似，没有任何地方引用该常量即可回收。
- **无用类回收**（满足三个条件）：
  1. 该类所有实例已被回收。
  2. 加载该类的`ClassLoader`已被回收。
  3. 该类对应的`java.lang.Class`对象没有任何地方被引用。
- 注意：方法区的回收不是必须的，需要虚拟机参数开启（如`-Xnoclassgc`）。

## 3. 垃圾回收算法

### 3.1 标记-清除算法
- **原理**：分为"标记"和"清除"两个阶段。先标记所有存活对象，然后统一回收所有未被标记的对象。
- **缺点**：
  - 效率不高（标记和清除两个过程都需要遍历）。
  - 产生大量不连续的内存碎片，导致后续大对象分配困难。
- **应用**：CMS收集器的老年代回收。

### 3.2 复制算法
- **原理**：将内存分为两块大小相等的区域，每次只使用其中一块。当这一块用完时，将存活对象复制到另一块，然后一次性清理当前块。
- **优点**：实现简单，运行高效，无内存碎片。
- **缺点**：内存利用率低（只有一半内存可用）。
- **应用**：新生代的Serial、ParNew等收集器（Eden与Survivor的8:1:1设计正是对复制算法的优化，只有10%的空间闲置）。

### 3.3 标记-整理算法
- **原理**：标记阶段与标记-清除相同，但后续不是直接清理，而是将所有存活对象向内存的一端移动，然后清理边界以外的内存。
- **优点**：无内存碎片，适合老年代。
- **缺点**：移动对象需要STW（Stop-The-World）。
- **应用**：Serial Old、Parallel Old等收集器。

### 3.4 分代收集算法
- **原理**：根据对象存活周期将内存划分为几块，不同区域使用不同算法：
  - **新生代**：对象"朝生夕死"，存活率低，使用**复制算法**。
  - **老年代**：对象存活率高，无额外空间进行分配担保，使用**标记-清除**或**标记-整理**算法。

### 3.5 三色标记算法
- **背景**：用于解决并发垃圾回收中的对象标记问题，特别是CMS和G1中的并发标记阶段。
- **三种颜色**：
  - **白色**：未被扫描的对象，若标记结束后仍为白色则不可达。
  - **灰色**：已被扫描但对象内的引用尚未被扫描完。
  - **黑色**：已被扫描且所有引用也已扫描完。
- **并发标记的问题**：
  - **漏标**：黑色对象指向了白色对象，同时灰色对象到该白色对象的引用被删除，导致白色对象被错误回收。
  - **多标**：原本要死亡的对象被误标为存活（浮动垃圾）。
- **解决方案**：
  - **增量更新（CMS）**：当黑色对象插入指向白色对象的新引用时，记录该引用，重新标记阶段再处理。
  - **原始快照（G1）**：当灰色对象删除指向白色对象的引用时，记录该引用，保证后续仍能扫描到。

## 4. 垃圾收集器

### 4.1 垃圾收集器分类
- **串行收集器**：Serial、Serial Old
- **并行收集器**：ParNew、Parallel Scavenge、Parallel Old
- **并发收集器**：CMS、G1、ZGC、Shenandoah

### 4.2 常见垃圾收集器详解

#### 4.2.1 Serial / Serial Old
- **特点**：单线程收集器，进行垃圾回收时必须暂停所有工作线程（Stop-The-World）。
- **适用场景**：客户端模式、单CPU环境、堆内存较小的情况。

#### 4.2.2 ParNew
- **特点**：Serial的多线程版本，使用多线程进行新生代垃圾回收。
- **适用场景**：与CMS配合使用（-XX:+UseConcMarkSweepGC时会自动启用ParNew）。

#### 4.2.3 Parallel Scavenge / Parallel Old
- **特点**：
  - **关注点**：吞吐量（CPU用于运行用户代码的时间与CPU总消耗时间的比值）。
  - **自适应调节**：`-XX:+UseAdaptiveSizePolicy`，虚拟机会根据当前系统运行情况动态调整参数。
- **适用场景**：后台运算、批处理等不需要太多交互的任务。
- **JDK 8默认收集器**：新生代Parallel Scavenge + 老年代Parallel Old。

#### 4.2.4 CMS（Concurrent Mark Sweep）
- **特点**：以获取最短回收停顿时间为目标的收集器，第一次实现垃圾收集线程与用户线程并发工作。
- **运作步骤**：
  1. **初始标记**（STW）：标记GC Roots能直接关联到的对象，速度快。
  2. **并发标记**：从GC Roots直接关联对象开始遍历整个对象图，耗时较长但可与用户线程并发。
  3. **重新标记**（STW）：修正并发标记期间因用户线程继续运行而产生变动的标记记录。
  4. **并发清理**：清理标记阶段判定为死亡的对象。
- **缺点**：
  - 对CPU资源敏感，并发阶段会占用CPU导致应用变慢。
  - 无法处理**浮动垃圾**（并发清理阶段产生的新垃圾）。
  - 基于**标记-清除**算法，产生内存碎片，可能导致Full GC退化。
  - 当老年代内存达到阈值（默认92%），可能触发**CMS Foreground GC**（退化为Serial Old），导致长时间STW。
- **关键参数**：
  - `-XX:CMSInitiatingOccupancyFraction=75`：设置CMS触发阈值。
  - `-XX:+UseCMSCompactAtFullCollection`：Full GC后进行内存压缩整理。
  - `-XX:CMSFullGCsBeforeCompaction=2`：多少次Full GC后进行一次压缩。
  - `-XX:+CMSParallelInitialMarkEnabled`：初始标记多线程。
  - `-XX:+CMSParallelRemarkEnabled`：重新标记多线程。

#### 4.2.5 G1（Garbage First）收集器
- **特点**：面向服务端，将Java堆划分为多个大小相等的独立**Region**，保留分代概念但不再是物理隔离。可预测的停顿时间模型。
- **Region**：JVM最多2048个Region，每个Region大小根据堆大小自动计算（如堆4G，Region约2M）。
- **运作步骤**：
  1. **初始标记**（STW）：标记GC Roots能直接关联的对象。
  2. **并发标记**：从GC Roots开始进行可达性分析。
  3. **最终标记**（STW）：处理SATB（原始快照）记录的引用变化。
  4. **筛选回收**：对各Region的回收价值和成本进行排序，根据用户设定的停顿时间（`-XX:MaxGCPauseMillis`）制定回收计划，采用复制算法将存活对象复制到空闲Region中。
- **优点**：可预测停顿、内存整理（无碎片）、并发收集。
- **适用场景**：大内存（>4G）、对延迟有要求的服务端应用。
- **JDK 9+默认收集器**。

#### 4.2.6 ZGC
- **特点**：JDK 11引入的实验性收集器，JDK 15正式生产。基于Region，采用着色指针、读屏障等技术，几乎全程并发，STW时间极短（<10ms）。
- **适用场景**：超大堆（TB级）、超低延迟要求。

### 4.3 收集器选择与默认配置
| JDK版本 | 默认收集器组合 |
|--------|--------------|
| JDK 8 | Parallel Scavenge + Parallel Old  |
| JDK 9~JDK 22 | G1  |

## 5. 类加载机制

### 5.1 类加载过程
类从被加载到虚拟机内存中开始，到卸载出内存为止，整个生命周期包括七个阶段：
1. **加载**：
   - 通过类的全限定名获取定义此类的二进制字节流。
   - 将字节流所代表的静态存储结构转化为方法区的运行时数据结构。
   - 在内存中生成代表这个类的`java.lang.Class`对象，作为方法区这个类的各种数据的访问入口。
2. **验证**：确保Class文件的字节流中包含的信息符合虚拟机规范，不会危害虚拟机自身安全。
3. **准备**：为类变量（static变量）分配内存并设置类变量初始值（零值），如`public static int value = 123`，准备阶段后value为0而不是123。
4. **解析**：将常量池内的符号引用替换为直接引用。
5. **初始化**：执行类构造器`<clinit>()`方法，真正开始执行Java代码，为类变量赋程序设定的值。
6. **使用**
7. **卸载**

### 5.2 类加载器
- **启动类加载器（Bootstrap ClassLoader）**：加载`<JAVA_HOME>/lib`目录中的核心类库，无法被Java程序直接引用。
- **扩展类加载器（Extension ClassLoader）**：加载`<JAVA_HOME>/lib/ext`目录中的类库。
- **应用程序类加载器（Application ClassLoader）**：加载用户类路径（ClassPath）上的类，是程序中默认的类加载器。
- **自定义类加载器**：继承`ClassLoader`，重写`findClass()`方法。

### 5.3 双亲委派模型
- **工作流程**：当一个类加载器收到类加载请求时，它首先不会自己去尝试加载这个类，而是把这个请求委派给父类加载器去完成，每一层次的类加载器都是如此，因此所有的加载请求最终都应该传送到顶层的启动类加载器中。只有当父加载器反馈自己无法完成这个加载请求时，子加载器才会尝试自己去加载。
- **优点**：
  - 避免类的重复加载。
  - 保证Java核心API的类型安全（如防止用户自定义`java.lang.Object`类覆盖核心类）。

### 5.4 打破双亲委派
- **场景**：
  - **Tomcat**：每个Web应用需要有自己的类加载器，隔离不同应用的类，且支持热部署。
  - **OSGi**：实现模块化热部署。
  - **JDBC**：通过`Thread.currentThread().getContextClassLoader()`获取上下文类加载器，实现SPI机制。
- **实现**：重写`loadClass()`方法，不遵循父加载优先的策略。

## 6. JVM调优

### 6.1 调优目标与指标
- **核心目标**：以最少的资源消耗，获得高吞吐量或低延迟。
- **关键指标**：
  - **吞吐量**：用户代码运行时间 / （用户代码运行时间 + GC时间）。
  - **延迟**：GC停顿时间（Stop-The-World时间），影响P95/P99响应时间。
  - **GC频率**：Minor GC和Full GC发生的次数。
  - **内存占用**：堆内存使用情况。

### 6.2 调优方法论
1. **分析需求和瓶颈**：找出性能瓶颈（内存不足、频繁GC、CPU过高等）。
2. **选择合适的GC算法**：
   - 实时性要求高 → G1、ZGC
   - 批处理/后台任务 → Parallel GC
   - 小内存环境 → Serial GC
3. **调整关键参数**：堆大小、代比例、GC阈值等。
4. **监控和调试**：通过GC日志分析效果，逐步调整。
5. **线上灰度验证**：先在测试环境或小流量验证，确认效果后全量上线。

### 6.3 核心调优参数详解

#### 堆内存设置
```bash
-Xms4g -Xmx4g              # 堆初始和最大值，建议相等避免动态调整
-Xmn2g                     # 新生代大小
-XX:SurvivorRatio=8        # Eden:Survivor=8:1
-XX:NewRatio=2             # 新生代:老年代=1:2
```

#### 元空间设置
```bash
-XX:MetaspaceSize=256m     # 元空间初始大小
-XX:MaxMetaspaceSize=256m  # 元空间最大大小，避免动态扩容触发Full GC
```

#### GC日志记录
```bash
# JDK 8及以前
-XX:+PrintGCDetails -XX:+PrintGCDateStamps -Xloggc:/path/to/gc.log

# JDK 9+
-Xlog:gc*:file=/path/to/gc.log:time
```

```bash

```

#### 典型优化案例

**案例1：频繁Full GC问题排查**
- **现象**：灰度发布一台机器后频繁Full GC。
- **排查**：对比正常机器，发现元空间使用量异常高，使用`jcmd`查看加载的类，发现大量动态生成的`MethodAccessor`。
- **解决**：将`-XX:MetaspaceSize`从1500m调整为2048m，Full GC消失。
- **总结**：元空间太小导致频繁Full GC，调大即可，根本原因是动态类加载过多。

**案例2：CMS退化为Foreground GC**
- **现象**：高并发下P99响应时间出现周期性毛刺，长达2-3秒。
- **分析**：CMS并发标记期间，用户线程继续运行，产生浮动垃圾。当老年代达到阈值（默认92%）触发CMS时，如果并发模式失败，会退化为STW的Foreground GC。
- **优化**：
  ```bash
  -XX:CMSInitiatingOccupancyFraction=75    # 提前触发CMS
  -XX:+UseCMSInitiatingOccupancyOnly       # 仅使用设置的阈值
  -XX:+UseCMSCompactAtFullCollection       # Full GC后压缩
  -XX:CMSFullGCsBeforeCompaction=2         # 2次Full GC后压缩一次
  ```
- **效果**：毛刺消失，系统响应稳定。

### 7.3 问题排查实战

#### 场景1：内存溢出（OOM）
**现象**：系统抛出`OutOfMemoryError: Java heap space`。
**排查步骤**：
1. **启动参数配置**：提前配置`-XX:+HeapDumpOnOutOfMemoryError`，OOM时自动生成堆DUMP。
2. **分析堆DUMP**：使用MAT或VisualVM打开DUMP文件。
3. **定位问题**：
   - 查看Leak Suspects，找到占用内存最大的对象。
   - 追踪引用链，找到GC Roots到该对象的路径。
   - 分析业务代码，确认是否因集合无限增长、缓存过大、连接未释放导致。
4. **解决**：修复代码或调整堆大小。

**案例**：618大促时某服务OOM，MAT分析发现`ThreadPoolExecutor`的队列中堆积了883个任务，每个任务包含一个6-7MB的大对象DTO，导致内存溢出。解决：增大老年代空间，优化大对象封装。

#### 场景2：CPU飙升
**现象**：系统CPU使用率持续高位，接口响应变慢。
**排查步骤**：
1. `top`或`htop`找到CPU高的Java进程PID。
2. `top -Hp <pid>` 找到CPU高的线程ID（十进制）。
3. 将线程ID转换为十六进制：`printf "%x\n" <tid>`。
4. `jstack <pid> > threaddump.txt`，搜索十六进制线程ID，查看该线程的堆栈。
5. 分析堆栈：可能是频繁GC（GC线程占用）、死循环、锁竞争等。
6. 如果是GC线程占用高，进一步分析GC日志，判断是否频繁Full GC或GC时间过长。

#### 场景3：死锁
**现象**：系统卡顿，部分请求无响应。
**排查**：
- `jstack <pid>`，如果存在死锁，输出末尾会显示"Found one Java-level deadlock"，并指出死锁线程和锁信息。
